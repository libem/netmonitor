# 网络状态监测工具

## 工具说明

该工具定时监测系统中的多个网络接口状态，通过 `ping` 指定目标地址，采集每个接口的可达率、丢包率和平均时延，并据此评估当前更优的网络链路。在 Linux 环境下，程序会通过修改默认路由的方式，将业务流量切换到评分更高的网络接口。

当前实现适用于类似 1.4G 专网、4G/5G 专网、有线网络等多链路场景。

## 主要功能

- 定时监测网络状态，需要监测的网络接口以及目标 IP/域名在 `config/monitor-net.yaml` 中定义
- 基于可达率、丢包率、平均时延计算接口评分，选出当前优先网络
- 判断当前默认路由接口是否需要切换
- 在 Linux 环境下通过设置默认路由快速切换网络
- 输出每轮探测结果、评分排名、切换原因和保留当前链路的原因，便于排障与调优

## 配置说明

示例配置：

```yaml
ping_targets:
  - "8.8.8.8"
  - "www.baidu.com"
ping_interval_sec: 10
ping_timeout_sec: 10
ping_count: 5
check_interval: 10s
interfaces:
  - "wwan0"
  - "lan0"
```

字段说明：

- `ping_targets`：每轮探测时要访问的目标地址列表，建议同时配置公网 IP、域名或业务侧关键地址
- `ping_interval_sec`：预留字段，当前版本未单独使用，默认值为 `10`
- `ping_timeout_sec`：单次 `ping` 命令超时时间
- `ping_count`：每次对目标发送的 `ping` 包数量
- `check_interval`：整轮检查的时间间隔
- `interfaces`：需要监控和参与评分的网络接口名称，必须与系统真实网卡名一致

## 工作流程

每轮检查的执行顺序如下：

1. 读取配置中的目标地址和网卡列表
2. 过滤系统中不存在的网卡
3. 对每个网卡依次 `ping` 所有目标地址
4. 汇总每个网卡本轮的探测结果并计算评分
5. 按评分从高到低排序，选出本轮最佳候选接口
6. 获取当前系统默认路由所在接口
7. 根据切换决策规则，判断是否需要切换默认路由
8. 输出日志，记录评分排名、切换原因或保留原因

## 评分算法说明

评分逻辑位于 `internal/evaluator/evaluator.go`。

### 1. 输入数据

每个网络接口在一轮检查中会得到多个 `PingResult` 样本，每个样本对应一个探测目标，主要包括：

- `Success`：该目标是否可达
- `PacketLoss`：该目标的丢包率，单位 `%`
- `AverageRTTMS`：该目标的平均往返时延，单位 `ms`

### 2. 聚合指标

程序会先把一个接口对多个目标的探测结果聚合为三个核心指标：

- 可达率 `Reachability`
  - 公式：`成功目标数 / 目标总数`
  - 取值范围：`0 ~ 1`
  - 说明：反映该接口当前“能否稳定打通外部目标”

- 平均丢包率 `PacketLoss`
  - 公式：`所有目标丢包率之和 / 目标总数`
  - 取值范围：理论上 `0 ~ 100`
  - 说明：反映链路稳定性，丢包越高越差

- 平均时延 `AverageRTTMS`
  - 公式：`所有成功采集到 RTT 的目标平均值`
  - 说明：反映链路响应速度，越低越好
  - 特殊处理：如果本轮没有采集到任何 RTT，则将其视为 `9999ms`，即极差时延，避免该接口被误判为优质链路

### 3. 评分公式

当前评分公式为：

```text
Score = Reachability * 100 - min(AverageRTTMS, 1000) / 10 - PacketLoss * 0.7
```

拆解说明：

- `Reachability * 100`
  - 可达率是最重要的信号
  - 如果一个接口能打通全部目标，则这一项最多贡献 `100` 分
  - 如果只能打通一半目标，则这一项只有 `50` 分

- `min(AverageRTTMS, 1000) / 10`
  - 时延作为惩罚项处理，时延越高扣分越多
  - 最大只按 `1000ms` 计入，防止极端高时延把整体分数无限拉低
  - 例如：`20ms` 会扣 `2` 分，`300ms` 会扣 `30` 分

- `PacketLoss * 0.7`
  - 丢包也作为惩罚项处理
  - 例如：`10%` 丢包会扣 `7` 分，`50%` 丢包会扣 `35` 分

### 4. 评分设计思想

当前算法体现的是一个偏保守的网络质量判断原则：

1. 先看“通不通”
   - 不可达的链路即使偶尔 RTT 很低，也不应被优先选择
2. 再看“稳不稳”
   - 丢包高的链路通常业务体验更差，应降低优先级
3. 最后看“快不快”
   - 在都可用的前提下，优先选择时延更低的链路

因此，可达率权重最高，其次是丢包和时延。

## 切换决策说明

切换逻辑同样位于 `internal/evaluator/evaluator.go`，主流程在 `internal/app/app.go` 中调用。

### 1. 候选接口选取

每轮评分完成后，程序会将所有接口按分数从高到低排序，排序第一名记为 `best`，作为本轮最优候选接口。

### 2. 当前接口识别

程序会查询当前默认路由绑定的接口，并尝试在本轮评分结果中找到该接口：

- 若找到，说明当前链路也在监控集合中，可以直接对比 `best` 与 `current`
- 若找不到，说明当前默认路由可能已经指向未纳入监控的网卡，程序会优先切换到本轮最优候选接口

### 3. 切换规则

程序按以下顺序判断是否切换：

#### 规则一：没有候选接口，不切换

- 条件：`best.Name == ""`
- 解释：当前没有有效评分结果，不能贸然切换

#### 规则二：最佳候选完全不可达，不切换

- 条件：`best.Reachability == 0`
- 解释：即使它在排序中排第一，只要一个目标都打不通，就不允许切换到它

#### 规则三：当前接口已经是最佳接口，不切换

- 条件：`best.Name == current.Name`
- 解释：当前链路已经是本轮最优，没有必要重复切换

#### 规则四：当前默认接口不在监控集合内，切换到最佳候选

- 条件：`currentTracked == false`
- 解释：说明当前默认路由未落在受控网卡中，为了让路由回到监控范围，直接切到 `best`

#### 规则五：分差不足阈值，不切换

- 条件：`best.Score - current.Score < MinSwitchScoreDelta`
- 当前阈值：`5`
- 解释：如果两个链路差距不明显，保持现状，避免因为轻微抖动导致频繁切换

#### 规则六：分差达到阈值，执行切换

- 条件：`best.Score - current.Score >= MinSwitchScoreDelta`
- 解释：说明候选链路相比当前链路确实明显更优，此时执行默认路由切换

### 4. 防抖思想

`MinSwitchScoreDelta = 5` 是当前版本的核心防抖参数。

它的作用是：

- 避免两个质量接近的链路在相邻周期中反复抢占默认路由
- 降低短时网络抖动、偶发丢包、瞬时 RTT 波动带来的频繁切换
- 保持链路切换动作更克制、更稳定

如果后续希望更保守，可以适当提高该值；如果希望更敏感，可以降低该值。

## 日志说明

程序每轮会输出三类关键日志：

- 单目标探测结果
  - 示例：`iface=lan0 target=8.8.8.8 success loss=0.00% avg_rtt=18.50ms`
- 单接口汇总结果
  - 示例：`iface=lan0 summary score=91.20 reachability=1.00 loss=0.00% avg_rtt=8.80ms`
- 排名与切换结论
  - 示例：`score ranking: lan0(91.20) > wwan0(73.40)`
  - 示例：`keep current interface=wwan0 reason=score delta 3.20 is below switch threshold 5 ...`
  - 示例：`switched default interface from wwan0 to lan0 reason=lan0 outperforms wwan0 by 17.80 points`

这些日志有助于判断：

- 某个目标是否对某块网卡不可达
- 当前评分是如何得出的
- 为什么程序没有切换
- 切换动作是由哪条规则触发的

日志会同时输出到标准输出和本地日志文件，默认策略如下：

- 日志目录：`logs/`
- 日志文件命名：`net-monitor-YYYYMMDD.log`
- 按天切分：跨天后自动写入新的日期文件
- 按大小切分：单个日志文件达到 `10MB` 后自动滚动为 `net-monitor-YYYYMMDD.1.log`、`net-monitor-YYYYMMDD.2.log`
- 自动清理：定期删除 `5` 天前的历史日志文件

## 调参建议

当前版本里最关键的调参点有两类：

- 评分公式中的权重
- 切换决策中的阈值

建议调参时遵循一个原则：先观察日志，再改参数，不要一开始就频繁修改多项参数，否则很难判断到底是哪一项起了作用。

### 1. 什么时候需要调参

如果你在现场运行时出现以下现象，就说明可以考虑调参：

- 明明某条链路业务体验更差，但评分仍然偏高
- 网络质量有轻微波动时，默认路由频繁来回切换
- 某链路延时明显更高，但因为可达率都为 `1`，分数差距不够明显
- 某链路偶发少量丢包，却被算法过度惩罚
- 现场最关心的是“稳定不断线”，而不是“绝对最低时延”

### 2. 评分公式调参建议

当前公式为：

```text
Score = Reachability * 100 - min(AverageRTTMS, 1000) / 10 - PacketLoss * 0.7
```

这意味着：

- 可达率满分贡献 `100`
- RTT 每增加 `10ms`，大约多扣 `1` 分
- 丢包率每增加 `1%`，大约多扣 `0.7` 分

#### 场景一：专网链路最重要的是“稳定在线”

适合场景：

- 工业设备、车载设备、边缘网关
- 业务对断线非常敏感
- 允许时延略高，但不能频繁丢包或不可达

建议：

- 保持甚至进一步提高可达率权重
- 提高丢包惩罚
- 适当降低时延惩罚

调参方向示例：

```text
Score = Reachability * 100 - min(AverageRTTMS, 1000) / 15 - PacketLoss * 1.0
```

效果：

- 更偏向选择“稳但稍慢”的链路
- 有丢包的链路会更快失去优先级

#### 场景二：业务更关注响应速度

适合场景：

- 控制指令、低时延交互、实时请求
- 两条链路都比较稳定，但 RTT 差异较大

建议：

- 提高时延惩罚
- 保持可达率优先，但让低时延链路更容易胜出

调参方向示例：

```text
Score = Reachability * 100 - min(AverageRTTMS, 1000) / 5 - PacketLoss * 0.7
```

效果：

- RTT 差异会更明显地体现在评分里
- 低时延链路更容易成为首选

#### 场景三：链路偶发少量丢包，但整体仍可接受

适合场景：

- 无线链路天然会有轻微抖动
- 不希望因为少量瞬时丢包而过度切换

建议：

- 降低丢包惩罚系数
- 同时适当提高切换阈值，减少敏感切换

调参方向示例：

```text
Score = Reachability * 100 - min(AverageRTTMS, 1000) / 10 - PacketLoss * 0.4
```

效果：

- 少量丢包不再造成过大分差
- 更适合波动型无线网络

### 3. 切换阈值调参建议

切换阈值由 `MinSwitchScoreDelta` 控制，当前值为：

```go
const MinSwitchScoreDelta = 5
```

它决定了：新链路即使排名第一，也必须比分别当前链路高出足够多，才会真正触发切换。

#### 阈值调大

例如改成 `8`、`10`：

- 更稳重
- 更不容易抖动切换
- 适合链路本身波动较大、但切换代价也较高的场景

适合：

- 专网切换后需要重新建连
- 设备端不希望频繁变更出口路由
- 两条链路质量通常接近

#### 阈值调小

例如改成 `2`、`3`：

- 更敏感
- 更容易追随当前最优链路
- 但也更容易产生抖动切换

适合：

- 对时延变化很敏感
- 切换代价很低
- 希望尽可能追求实时最优出口

### 4. 推荐调参方法

建议按照下面顺序逐步调整：

1. 先保持评分公式不动，只调整 `MinSwitchScoreDelta`
   - 先看当前问题是不是“切换太频繁”
   - 如果是，优先增大阈值
2. 再根据日志判断到底是 RTT 还是丢包导致排序不符合预期
   - 如果慢链路经常赢，增强 RTT 惩罚
   - 如果不稳定链路经常赢，增强丢包惩罚
3. 每次只改一个参数，并至少观察多个检查周期
   - 这样更容易定位改动效果

### 5. 实战建议

- 如果现场网络波动大，先调切换阈值，再动评分公式
- 如果现场网络很稳定，但总是选不到“更快”的链路，优先增强 RTT 惩罚
- 如果业务最怕断流，优先增强丢包惩罚，并保守提高切换阈值
- 如果后续发现不同目标地址的重要性不同，可以进一步扩展为“目标加权评分”
- 如果后续发现单轮结果抖动明显，可以增加“多轮滑动平均”或“连续多轮胜出才切换”的机制


## systemd 开机自启动配置

如果目标 Linux 系统使用 `systemd`，可以使用仓库中的示例服务文件：`deploy/systemd/net-monitor.service`。

该服务具备以下特点：

- 使用 `After=network-online.target` 与 `Wants=network-online.target`，尽量等待网络就绪后再启动
- 以后台常驻方式运行
- 进程异常退出后自动拉起
- 允许程序执行 `ping` 和路由切换所需的网络能力

### 1. 准备程序目录

建议在目标机上准备如下目录：

```text
/opt/net-monitor/
├── net-monitor
├── config/
│   └── monitor-net.yaml
└── README.md
```

其中：

- `net-monitor` 为编译后的可执行文件
- `config/monitor-net.yaml` 为运行配置文件

例如在当前项目中可先构建 ARM64 版本：

```bash
make build-linux-arm64
```

然后将 `bin/net-monitor-arm` 重命名为 `net-monitor` 并拷贝到目标目录。

### 2. 安装 service 文件

将服务文件复制到系统目录：

```bash
sudo cp deploy/systemd/net-monitor.service /etc/systemd/system/net-monitor.service
```

如果你的安装目录不是 `/opt/net-monitor`，请同步修改 service 文件中的：

- `WorkingDirectory`
- `ExecStart`
- `Environment=NETMONITOR_CONFIG=...`
- `Documentation`

### 3. 重新加载并设置开机启动

```bash
sudo systemctl daemon-reload
sudo systemctl enable net-monitor.service
sudo systemctl start net-monitor.service
```

### 4. 查看运行状态

```bash
sudo systemctl status net-monitor.service
sudo journalctl -u net-monitor.service -f
```

### 5. 注意事项

- 当前 service 示例默认使用 `root` 运行，这是因为路由切换通常需要足够权限
- 如果你已经通过其他方式授予程序 `CAP_NET_ADMIN`、`CAP_NET_RAW`，也可以再按现场需要调整运行用户
- 程序本身还会在工作目录下生成 `logs/` 日志文件，请确认 service 的工作目录可写
- `network-online.target` 能提升“网络就绪后启动”的概率，但并不保证业务网络一定可达；真正链路质量仍由程序自身持续探测判断

## 使用注意事项

- 当前默认路由切换实现依赖 Linux 的 `ip route` 命令
- 执行路由切换通常需要 root 权限或具备相应网络管理权限
- `interfaces` 中的网卡名称必须与目标主机实际网卡名一致
- `ping_targets` 建议选择稳定、可代表业务连通性的目标地址
- 当前评分权重和切换阈值为经验值，后续可根据现场网络特性进一步调优
